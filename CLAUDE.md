## 절대 규칙 (어기면 안 됨)

1. **오프라인 단일 파일 동작**: 각 도구는 인터넷 없이 HTML 파일 하나만 열어도 동작해야 한다.
   외부 스크립트·API·CDN 의존을 새로 추가하지 않는다(구글 폰트 링크만 예외 — 없어도 동작에
   지장이 없어야 함).
2. **일지 생성기(pgrecord) 수정 경로**: 문장 데이터는 `pgrecord/문장은행.json`, 로직·UI는
   `pgrecord/template.html`에서 고친 뒤 `build.py`로 빌드해 산출물을 `index.html`에 반영한다.
   **`pgrecord/index.html`을 직접 편집하지 않는다.**
3. **일지 문체**: 생성 문장은 `~하심 / ~하셨음` 공문서체를 유지한다. 같은 회차 안에서 같은
   문장이 중복 배분되면 안 된다.
4. **개인정보 금지**: 이용 어르신의 실명·등급 등 개인정보를 코드·데이터·커밋에 넣지 않는다.
5. **기존 데이터 보존**: pg2처럼 localStorage를 쓰는 도구는 수정 후에도 기존 저장 데이터가
   그대로 읽혀야 한다. 저장 형식을 바꿔야 하면 이전 형식을 읽는 변환 코드를 함께 넣는다.

## 작업 순서 (모든 요청 공통)

1. 요청이 모호하면 질문으로 멈추지 말고, 합리적인 가정을 한 줄로 밝힌 뒤 그 가정대로 작업을
   끝까지 진행한다. 판단이 갈리는 지점만 마지막에 질문한다.
2. 수정 전에 관련 코드를 실제로 읽고 현재 동작을 파악한다. 추측으로 고치지 않는다.
3. 레이아웃·인쇄 관련 수정은 A4 인쇄 화면과 모바일 화면을 함께 확인한다.
4. 완료 보고는 아래 형식으로 한다:
   - 무엇이 왜 바뀌었는지 2~3줄
   - 직원에게 안내할 한 줄 문구 (예: "이제 요일 제목을 누르면 그 요일 전체가 채워집니다")
5. **진행 확인을 묻지 않는다**: 완결된 변경은 커밋·푸시하고, 필요하면 PR을 만들어 머지까지
   자율적으로 끝낸다("머지할까요?", "PR 만들까요?" 같은 확인을 구하지 않는다). 되돌리기 어렵거나
   판단이 갈리는 지점만 예외로 질문한다.

## 버전·커밋 규칙

- 버전 표기: `v{YYYY.MM.DD.NNN}` — 그날의 작업 순번. 같은 날 두 번째 작업이면 `.002`.
- 버전을 올릴 때 HTML `<title>` 아래 주석과 헤더의 `.ver` 표기를 **둘 다** 갱신한다.
- 커밋 메시지: `도구명 v버전 — 변경 요지 한 줄` (한국어).
  예: `프로그램 일지 생성기 v2026.07.06.007 — 총평 기반 생성으로 전환`
- 커밋 하나 = 완결된 변경 하나. 서로 다른 도구의 수정을 한 커밋에 섞지 않는다.

## 새 도구를 추가할 때

- 폴더 하나 = 도구 하나. 폴더 안에 직원용 사용법 `README.md`를 만들고, 루트 `README.md`의
  표에 링크를 추가한다.
- **프로그램·인지학습 관련 도구는 `pg/` 폴더 하위에 만든다**(예: `pg/새이름/`). README·도구
  목록에서도 "프로그램 도구" 묶음 안에, "그 밖의 도구"보다 위에 넣는다. 프로그램과 무관한
  도구만 `pg/` 밖 별도 폴더로 만든다.
- 기존 디자인 토큰을 재사용한다: `--navy #1F3864`, `--teal #1D9E75`, Noto Sans KR, 카드형
  레이아웃, 상단 남색 헤더.
- `<head>`에 사이트 아이콘을 넣는다: `<link rel="icon" type="image/svg+xml" href="…/favicon.svg">`.
  경로는 폴더 깊이에 맞춘 상대경로(루트 하위 1단계=`../favicon.svg`, 2단계=`../../favicon.svg`).
  아이콘 원본은 루트 `favicon.svg`(남색 라운드 + 흰 W + teal 밑줄)이며 새 도구도 같은 아이콘으로 통일한다.
- 처음부터 위의 절대 규칙(오프라인 동작, 개인정보 금지, 버전 표기)을 적용한다.
