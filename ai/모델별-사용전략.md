# 모델별 사용 전략 — 모델이 바뀌어도 품질을 유지하는 법

원칙은 하나다: **지침(무엇을/어떤 톤으로)은 모든 모델에 동일하게 넣고, 모델이 가벼울수록
"구조와 예시"를 더 많이 준다.** 지침을 모델마다 다시 쓰지 않는다.

## 공통 (모든 모델)

- `공통-커스텀-인스트럭션.md`를 그 모델의 "영구 지침" 자리에 한 번 넣는다.
  - ChatGPT: 설정 → Custom Instructions, 또는 프로젝트(Projects) 지침
  - Gemini: Gems 만들기 → 지침 칸
  - Claude(웹/앱): 프로젝트 → 프로젝트 지침
  - Claude Code / Cursor: 저장소의 `CLAUDE.md` / `.cursorrules`에 병합
- 잘 나온 결과물은 "예시 은행" 폴더(노션 또는 이 저장소 `ai/예시/`)에 저장해 두고,
  새 작업 때 가장 비슷한 예시 1개를 붙여 넣는다. **예시 1개가 지침 열 줄보다 강하다.**

## 상위 모델 (Claude Fable/Opus, GPT 최상위)

- 설계·컨설팅·긴 문서·여러 산출물 동시 작업에 쓴다.
- "초안 → 스스로 비판 → 수정본까지 한 번에"를 요청해도 소화한다.
- 모호한 요청을 던져도 가정을 잘 세우므로, 배경 설명을 길게 쓰는 대신 목적만 분명히 준다.

## 중간 모델 (Claude Sonnet, Gemini Pro, GPT 표준)

- 일상 콘텐츠 제작의 주력. 품질은 모델이 아니라 **템플릿 밀도**가 결정한다.
- 요청할 때 형식 기본형(릴스/카드뉴스/블로그 구조)을 명시하거나 스킬을 지정한다.
- 예시 1개를 같이 주면 상위 모델과 체감 차이가 거의 없어진다.

## 경량 모델 (Claude Haiku, Gemini Flash, GPT mini)

- 한 번에 산출물 1개만 시킨다. "카드뉴스 8장 + 릴스 대본"처럼 묶지 않는다.
- 단계를 쪼갠다: ① 후킹 5안 → ② 고른 후킹으로 본문 → ③ 검수. 한 메시지 = 한 단계.
- 출력 형식을 빈 틀로 직접 준다(장 번호까지 적어서). 자유도를 줄일수록 품질이 오른다.
- 검증 판단이 필요한 내용(수가, 제도)은 경량 모델에 맡기지 않는다.

## 도구형 (Claude Code, Cursor)

- 코드·도구 작업은 반드시 저장소 지침(`CLAUDE.md`)이 있는 상태에서 시작한다.
- 반복 작업은 스킬(`.claude/skills/`)로 만들어 "지난번처럼"이 통하게 한다.
- 커밋 메시지·버전 규칙은 CLAUDE.md에 있으므로 매번 다시 설명하지 않는다.

## 모델을 갈아탈 때 점검 3가지

1. 영구 지침 자리에 공통 인스트럭션이 들어갔는가
2. 예시 1개를 첨부했는가
3. 경량 모델이면 작업을 단계로 쪼갰는가
